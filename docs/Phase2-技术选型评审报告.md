## 2. 浏览器自动化引擎选型

### 2.1 候选方案对比

| 特性 | Playwright | Selenium | Puppeteer |
|------|-----------|----------|-----------|
| **浏览器支持** | Chromium, Firefox, WebKit | 所有主流浏览器 | 仅Chromium |
| **API风格** | 现代异步API | 同步/异步混合 | 异步API |
| **性能** | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐⭐ |
| **稳定性** | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐⭐ |
| **文档质量** | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐⭐ |
| **社区活跃度** | 高 | 非常高 | 高 |
| **学习曲线** | 低 | 中 | 低 |
| **Python支持** | 原生支持 | 原生支持 | 需要pyppeteer |
| **无头模式** | 支持 | 支持 | 支持 |
| **网络拦截** | 强大 | 有限 | 强大 |
| **等待机制** | 智能等待 | 显式等待 | 智能等待 |
| **资源占用** | 中 | 高 | 中 |

### 2.2 选型决策

**最终选择**: Playwright

**理由**:
1. **现代化API**: 异步API设计优秀，与FastAPI完美配合
2. **性能优秀**: 启动速度快，资源占用合理
3. **稳定可靠**: 智能等待机制，减少flaky测试
4. **功能强大**: 支持网络拦截、请求修改、截图等高级功能
5. **多浏览器支持**: 虽然主要使用Chromium，但保留了扩展性
6. **活跃维护**: Microsoft官方维护，更新频繁

**不选Selenium的原因**:
- API较老，同步风格不适合异步架构
- 性能相对较差
- 配置复杂度高

**不选Puppeteer的原因**:
- 仅支持Chromium，扩展性受限
- Python支持需要第三方库pyppeteer，维护不如官方

### 2.3 风险评估

| 风险 | 等级 | 缓解措施 |
|------|------|----------|
| 浏览器资源占用过高 | 中 | 限制并发数，使用无头模式，复用浏览器实例 |
| 目标网站反爬虫 | 中 | User-Agent伪装，控制访问频率，支持代理 |
| 版本兼容性问题 | 低 | 锁定Playwright版本，定期更新测试 |

**评审结论**: ✅ 通过，建议采用Playwright

## 3. 任务调度系统选型

### 3.1 候选方案对比

| 特性 | APScheduler | Celery | Airflow | Prefect |
|------|-------------|--------|---------|---------|
| **复杂度** | 低 | 中 | 高 | 中 |
| **学习曲线** | 低 | 中 | 高 | 中 |
| **Cron支持** | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ |
| **异步支持** | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ |
| **持久化** | 支持 | 支持 | 支持 | 支持 |
| **分布式** | 有限 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ |
| **监控UI** | 无 | Flower | 内置 | 内置 |
| **依赖管理** | 无 | 无 | DAG | DAG |
| **资源占用** | 低 | 中 | 高 | 中 |
| **部署复杂度** | 低 | 中（需要消息队列） | 高 | 中 |

### 3.2 选型决策

**最终选择**: APScheduler

**理由**:
1. **轻量级**: 无需额外的消息队列，部署简单
2. **Cron支持**: 完整支持Cron表达式
3. **异步友好**: 原生支持asyncio，与FastAPI无缝集成
4. **持久化**: 支持数据库持久化，重启不丢失任务
5. **足够用**: 对于单机部署场景，功能完全满足需求

**不选Celery的原因**:
- 需要额外的消息队列（RabbitMQ/Redis）
- 配置复杂度高
- 对于当前需求过度设计

**不选Airflow的原因**:
- 过于重量级，适合复杂DAG工作流
- 学习曲线陡峭
- 资源占用高

**不选Prefect的原因**:
- 相对较新，生态不如APScheduler成熟
- 对于简单定时任务过度设计

### 3.3 扩展性考虑

如果未来需要分布式调度，可以考虑：
1. 使用Redis作为APScheduler的JobStore，实现多实例协调
2. 迁移到Celery（需要重构，但API相似）
3. 使用Kubernetes CronJob（云原生方案）

**评审结论**: ✅ 通过，建议采用APScheduler

## 4. 内容处理工具选型

### 4.1 HTML解析工具

**选择**: BeautifulSoup4

**理由**:
- Python生态最成熟的HTML解析库
- API简单易用
- 支持多种解析器（lxml, html.parser）
- 容错能力强

**替代方案**:
- lxml: 性能更好，但API复杂
- html5lib: 严格遵循HTML5标准，但速度慢

### 4.2 HTML转Markdown工具

**选择**: html2text

**理由**:
- 转换质量高
- 保留文档结构
- 配置灵活

**替代方案**:
- markdownify: 功能类似，但社区较小
- 自定义转换: 开发成本高

### 4.3 文本分块工具

**选择**: 复用现有RAG系统的分块逻辑

**参数**:
- chunk_size: 1000 tokens
- chunk_overlap: 200 tokens

**理由**:
- 与现有知识库系统保持一致
- 已经过验证，效果良好

**评审结论**: ✅ 通过

## 6. 部署和运维方案

### 6.1 容器化部署

**方案**: Docker + Docker Compose

**配置要点**:
```yaml
services:
  backend:
    build: ./backend
    environment:
      - PLAYWRIGHT_BROWSERS_PATH=/ms-playwright
    volumes:
      - playwright-cache:/ms-playwright
```

**Playwright安装**:
```dockerfile
RUN pip install playwright && \
    playwright install chromium && \
    playwright install-deps
```

### 6.2 资源限制

**建议配置**:
- CPU: 2核心（最低1核心）
- 内存: 4GB（最低2GB）
- 磁盘: 20GB（浏览器缓存占用约500MB）

**并发限制**:
- 最大并发采集任务: 5个
- 单个任务超时: 30分钟

### 6.3 监控方案

**日志监控**:
- 采集任务执行日志
- 错误日志和异常追踪
- 性能指标（执行时间、成功率）

**告警规则**:
- 连续3次采集失败
- 任务执行超时
- 浏览器进程异常

**评审结论**: ✅ 通过

## 8. 性能评估

### 8.1 预期性能指标

| 指标 | 目标值 | 说明 |
|------|--------|------|
| 单页采集时间 | < 10秒 | 包含页面加载和内容提取 |
| 并发采集数 | 5个 | 同时运行的采集任务 |
| 调度延迟 | < 10秒 | 定时任务触发延迟 |
| 内存占用 | < 500MB/任务 | 单个采集任务的内存占用 |
| CPU占用 | < 50% | 5个并发任务时的CPU占用 |

### 8.2 性能优化策略

**浏览器优化**:
- 使用无头模式
- 禁用不必要的资源加载（图片、CSS）
- 复用浏览器实例
- 设置合理的超时时间

**数据库优化**:
- 添加必要的索引
- 使用连接池
- 批量插入文档

**缓存优化**:
- Redis缓存已采集URL
- 避免重复采集

**评审结论**: ✅ 通过，性能目标合理

## 13. 对比分析：自研 vs 第三方服务

### 13.1 第三方服务对比

| 服务 | 优点 | 缺点 | 成本 |
|------|------|------|------|
| Scrapy Cloud | 成熟稳定 | 不支持JS渲染 | $9/月起 |

### 13.2 自研方案优势

**优势**:
1. **完全控制**: 可以根据需求定制功能
2. **成本低**: 无需支付第三方服务费用

4. **集成度高**: 与现有系统无缝集成
5. **无限制**: 无采集次数、并发数限制

**劣势**:
1. **开发成本**: 需要投入开发时间
2. **维护成本**: 需要持续维护
3. **功能完善度**: 初期功能相对简单
